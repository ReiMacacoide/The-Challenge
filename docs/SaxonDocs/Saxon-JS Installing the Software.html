Installing the Software

Saxon-JS consists entirely of JavaScript code which is downloaded from your web site to run in the user's browser. The JavaScript modules can therefore simply be placed alongside any necessary XML and HTML files in a static area of your web site, and referenced from a script element on the HTML page in the usual way.

Before you can use Saxon-JS you will need to compile an XSLT stylesheet into the export format which Saxon-JS recognizes. This requires Saxon-EE, which is a commercial Saxonica product for which you will need a license key. The development of the stylesheet can take place on a different machine from one where the web server is deployed: the exported stylesheet (SEF) is an XML document and can be freely moved to wherever it is needed. No license key is needed to run Saxon-JS itself. See Exporting for JS for further details.

The software is issued in a ZIP file Saxon-JS-1.X.zip. This includes both a readable source version of the code (SaxonJS.js) and a minified version (SaxonJS.min.js). The code all runs client-side: there are no run-time dependencies on the web server environment other than the ability to serve static content.

Also included in the Saxon-JS ZIP file are a number of data files, and the JavaScript module for dynamic XPath evaluation. These are read from the server only if they are actually needed. The files are:

  • opt/categories.json - contains lists of the Unicode characters in each Unicode block (for example Hebrew or Tagalog) and category (for example Lu or Nd). This data is used to expand regular expressions containing constructs such as \p{IsHebrew} or \p{Lu}.
  • opt/functions.json - contains a list of all the XPath functions implemented in Saxon-JS. This data is used in the rare event that function-available() is called with a dynamically-computed argument.
  • lib/xpath/XPathJS.min.js - add-on module for dynamic XPath evaluation using Saxon-JS. This module is dynamically loaded when the xsl:evaluate instruction is used.

The ZIP archive should simply be unzipped into an appropriate location in the web server's filestore layout, where the files can be referenced from HTML pages in the same way as other JavaScript components.
