Migration from Saxon-CE

This section lists some of the differences between Saxon-JS and Saxon-CE that are likely to affect applications which were originally built for use with Saxon-CE.

  • Stylesheet preparation: Saxon-JS requires the stylesheet to be compiled on the server using the Saxon Java product, creating a stylesheet export file which is referenced from the browser. This process must be repeated every time the stylesheet changes. Two consequences of this are:

    • It is not possible to create a stylesheet programmatically in the browser and then execute it.
    • It is not possible to access the stylesheet as a source document (use of document(""), which was a common coding pattern with XSLT 1.0, does not work). The source stylesheet can of course be mounted as a file on the web server in which case it can be accessed like any other source document, but it is not accessible using document("") with an empty URI.
  • Invocation: Saxon-CE provided a variety of mechanisms for invoking a transformation, designed in part to eliminate the need for writing any JavaScript. Saxon-JS is designed more on the assumption that XSLT and JavaScript code will co-exist, and the invocation API is therefore designed explicitly for use from JavaScript, typically in a page load event.
  • Pseudo-attributes in the prop: and prop:style namespaces (http://saxonica.com/ns/html-property and http://saxonica.com/ns/html-style-property) are not implemented. Instead, use ixsl:get() to get properties of a node, and ixsl:style() to get properties of its style.
  • Logging: The Saxon-CE features for logging execution of stylesheet constructs currently have no equivalent in Saxon-JS.
  • The xsl:result-document instruction does not support dynamic path expressions such as href="?select=.//tr/td[1]". However from Saxon-JS 0.9.1, the construct href="?." can be used to direct output to the current context item, so in combination with xsl:for-each this provides an alternative approach. Otherwise, output is directed to elements with specific id values, which can be constructed programmatically.
  • The construct match="ixsl:window()" for handling non-document events (such as a change to the contents of the URL fragment identifier) is not available in Saxon-JS. Use match="." instead.
  • The ixsl:call() extension function takes the arguments to be supplied to the called JavaScript function as a single array value, rather than using a variable number of arguments to ixsl:call().
  • Delayed update of result documents was implemented in Saxon-CE, and may appear in Saxon-JS in the future, but it is not implemented in this release of Saxon-JS.
